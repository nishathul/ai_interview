# Interview System Technical Implementation

## Overview

This project implements a local-first interview recording platform using:

- React frontend for candidate and admin UX
- Node.js control backend
- Janus WebRTC media server for real-time ingest and server-side recording
- Post-processing pipeline for conversion and audio/video merge

The solution is designed to support:

- shareable attendee-only interview links
- reliable server-side recording
- interview history and playback
- operational visibility (health + performance metrics)

## High-Level Architecture

```text
Candidate Browser (React)
  ├─ Capture: getUserMedia (camera + mic)
  ├─ Publish: WebRTC to Janus VideoRoom
  └─ UX: attendee mode + performance monitor

Node.js Backend (Express)
  ├─ Session lifecycle (create/end/delete)
  ├─ Janus room provisioning (VideoRoom plugin)
  ├─ Interview listing + metadata aggregation
  └─ Recording conversion triggers

Janus Gateway (Docker)
  ├─ Receives WebRTC publisher stream
  ├─ Records stream tracks as .mjr
  └─ Exposes HTTP/WS APIs used by frontend/backend

Conversion & Merge Pipeline
  ├─ janus-pp-rec: .mjr -> .webm (video) / .opus (audio)
  ├─ ffmpeg: video + audio -> merged .mp4
  └─ Host storage for playback and download
```

## Frontend Implementation

Primary file: `frontend/src/App.jsx`

UI modules:

- `frontend/src/components/SessionControls.jsx`
- `frontend/src/components/StatusPanel.jsx`
- `frontend/src/components/HealthPanel.jsx`
- `frontend/src/components/AdminPanel.jsx`
- `frontend/src/components/InterviewHistory.jsx`
- `frontend/src/components/AttendPerformancePanel.jsx`

### Modes

1. Normal mode (`http://localhost:5173`)
   - session creation
   - interview controls
   - admin filters/history/conversion/delete
   - backend/janus health

2. Attend mode (`mode=attend` in query string)
   - candidate-only interview controls
   - local preview
   - live performance monitor
   - admin and history hidden

### Session Creation & Link Sharing

- Frontend calls `POST /session/create`.
- Backend returns session payload including `start_link`.
- `start_link` embeds:
  - `mode=attend`
  - `session_id`
  - `room_id`
  - `token`
  - janus endpoint hints

Opening the link loads attendee mode and pre-populates session context from URL parameters.

### WebRTC Publishing Flow

1. `getUserMedia` captures audio/video.
2. Local preview attaches stream to `<video>` element.
3. Janus client initializes via npm package (`janus-gateway`).
4. Frontend attaches to `janus.plugin.videoroom`.
5. Publisher joins room and sends SDP offer.
6. Publish request includes `record: true` and `filename: session_id`.

### Reliability and Save Guarantees

Interview end persistence is protected by:

- explicit end call on "End Interview" (`POST /session/:sessionId/end`)
- unload/pagehide backup save using `navigator.sendBeacon`
- duplicate-write guard to avoid double-ending

This prevents losing interview end metadata when the attendee closes/reloads quickly.

### Performance Monitoring (Attend Page)

Displayed metrics combine browser network APIs, Janus callbacks, and WebRTC stats:

- online/offline state
- effective network type, RTT, downlink
- WebRTC/ICE/connection states
- media activity (audio/video on/off)
- outbound bitrate (kbps), fps, packet loss
- slow-link event count

Stats are polled while live and updated in real time.

## Backend Implementation

Primary file: `backend/src/server.js`

### Core Responsibilities

- Manage session metadata in `backend/data/sessions.json`
- Create Janus VideoRoom rooms for each interview
- Generate shareable attendee links
- End sessions and persist duration/status
- Trigger conversion pipeline
- Aggregate interview metadata with available files
- Provide health diagnostics

### Important Endpoints

- `GET /health` basic backend liveness
- `GET /health/details` backend + Janus connectivity check
- `POST /session/create` create room/session + attendee start link
- `POST /session/:sessionId/end` persist end status and duration
- `GET /sessions` raw session list
- `GET /interviews` sessions enriched with discovered recordings
- `DELETE /interviews/:sessionId` remove metadata and local files
- `POST /recordings/process` convert all
- `POST /recordings/process/:sessionId` convert specific interview

### Janus Room Provisioning

Backend creates a Janus session and attaches VideoRoom plugin, then issues:

- `request: "create"`
- `record: true`
- `rec_dir: "/recordings"`
- bitrate and keyframe controls for stability:
  - `bitrate`
  - `bitrate_cap`
  - `fir_freq`

## Janus and Media Storage

Janus runs via `docker-compose.yml`.

Key ports:

- HTTP API: `8088`
- WS API: `8188`
- RTP/RTCP UDP media range: `10000-10200`

Track outputs:

- raw track recordings (`.mjr`) are generated by Janus
- host-visible folders:
  - `infra/recordings` (raw + audio artifacts)
  - `storage` (video artifacts + merged playback files)

## Conversion and Merge Pipeline

Primary script: `scripts/convert-recordings.sh`

### Steps

1. Discover `.mjr` recordings in Janus container directories.
2. Convert each track using `janus-pp-rec`:
   - `*-video-*.mjr` -> `.webm`
   - `*-audio-*.mjr` -> `.opus`
3. Copy artifacts to host folders.
4. Merge audio + video into playback output:
   - preferred: host `ffmpeg`
   - fallback: Docker image `jrottenberg/ffmpeg:6.0-alpine`
5. Produce merged file:
   - `storage/<session_id>-merged.mp4`

The UI prefers merged outputs for playback.

## Dashboard Data Model

`GET /interviews` computes dashboard rows by combining:

- session metadata (`sessions.json`)
- raw files matched by `session_id`
- converted files matched by `session_id`

Each item includes:

- `session_id`, `candidate_id`, `room_id`, status timestamps/duration
- `raw_recordings[]`
- `converted_recordings[]`

## Development Operations

Start full local stack:

- `./scripts/dev-up.sh`

Stop full local stack:

- `./scripts/dev-down.sh`

These scripts manage Janus + frontend + backend processes and logs.

## Current Constraints

- Local file storage only (no object storage yet)
- Minimal auth enforcement (token generated but not deeply validated in Janus path)
- Single-node deployment model
- Session store is JSON file, not database-backed
